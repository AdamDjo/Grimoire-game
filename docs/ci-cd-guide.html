🤔 C'est quoi le CI/CD ?

Imagine que tu travailles avec 3 développeurs sur le même projet. Chacun code de son côté. Quand quelqu'un pousse du code, comment tu sais que ça ne casse pas ce que les autres ont fait ? C'est là qu'intervient le CI/CD.

CI — Intégration Continue

Continuous Integration — À chaque fois que quelqu'un pousse du code, une série de vérifications automatiques se lancent :

  • Le code compile-t-il ?
  • Les tests passent-ils ?
  • Le linting est-il respecté ?
  • Les types TypeScript sont-ils corrects ?
CD — Déploiement Continu

Continuous Deployment — Si le CI est vert, on peut automatiquement :

  • Créer une release
  • Publier un tag git
  • Déployer en production
  • Notifier l'équipe
💡
Analogie simple
Le CI/CD c'est comme un contrôle qualité en usine. Sur une chaîne de montage, chaque pièce passe par des tests automatiques avant d'aller à l'étape suivante. Si une pièce est défectueuse, la chaîne s'arrête. En code c'est pareil — si le lint échoue, le test ne se lance pas, si le test échoue, le build ne se lance pas.

🎯 Pourquoi en a-t-on besoin ?

Sans CI/CD, voici ce qui se passe dans une vraie équipe :

😤

Dev A pousse du code vendredi soir

Il n'a pas lancé les tests. "Ça marche sur ma machine."

😴

Dev B pull lundi matin

Son environnement est différent. Les tests qu'il lance échouent. Il perd 2h à comprendre pourquoi.

💥

Lundi après-midi : la prod est cassée

Personne ne sait quel commit a causé le bug. Il faut revenir en arrière manuellement.

✅
Avec le CI/CD
Le code du Dev A est rejeté automatiquement vendredi soir. GitHub bloque la PR. Dev A voit l'erreur immédiatement et la corrige. Dev B ne voit jamais ce code cassé. La prod reste stable.

Les bénéfices concrets

🛡️
Fiabilité

Le code qui atteint main a toujours passé tous les checks. Jamais de surprise.

⚡
Vitesse

Les reviews humaines se concentrent sur la logique, pas sur "as-tu lancé le lint ?".

📖
Traçabilité

Chaque commit a un statut vert ou rouge. On sait exactement quel commit a cassé quoi.

⚙️ GitHub Actions — L'outil

GitHub Actions est le moteur qui fait tourner notre CI/CD. C'est un service directement intégré à GitHub — pas besoin d'un serveur externe.

Les concepts clés

📄 Workflow

Un fichier YAML dans .github/workflows/. C'est la recette complète : quand lancer, quoi faire.

ci.yml pr.yml release.yml
🎯 Trigger (déclencheur)

Ce qui lance le workflow. Exemples :

  • push — quelqu'un pousse du code
  • pull_request — une PR est ouverte
  • schedule — tâche planifiée (cron)
💼 Job

Un ensemble de steps qui tournent sur une machine virtuelle (ubuntu). Les jobs peuvent tourner en parallèle ou en séquence avec needs:.

🔧 Step

Une action atomique dans un job. Soit une commande bash (run:), soit une action réutilisable (uses:).

Structure d'un workflow

.github/workflows/ci.yml 
# 1. NOM du workflow (affiché dans GitHub Actions)
name: CI

# 2. TRIGGER — quand ce workflow se déclenche
on:
  push:
    branches: [main, develop, "release/*"]
  pull_request:
    branches: [main, develop]

# 3. JOBS — ce qui se passe
jobs:
  lint-and-typecheck:          # nom du job
    runs-on: ubuntu-latest    # machine virtuelle GitHub
    steps:
      - uses: actions/checkout@v4   # clone le repo
      - name: Install pnpm
        run: npm install -g pnpm@9.15.0
      - name: Lint
        run: pnpm lint

  test:
    needs: lint-and-typecheck    # attend que lint soit vert
    runs-on: ubuntu-latest
    steps:
      - run: pnpm test

La machine virtuelle GitHub

🖥️
Chaque job = une machine fraîche
GitHub crée une nouvelle machine Ubuntu vierge pour chaque job. Elle installe Node, pnpm, clone ton repo, lance les commandes, puis est détruite. C'est pour ça qu'on doit réinstaller pnpm à chaque job — aucune donnée n'est partagée entre les jobs par défaut.

L'ordre a de l'importance — le principe du "fail fast"

On utilise needs: pour que les jobs s'enchaînent dans l'ordre. Pourquoi ? Pour échouer le plus tôt possible.

Exemple concret

Tu as une faute de frappe dans une variable TypeScript. Le lint va échouer en 30 secondes. Sans needs:, les tests ET le build se lancent quand même en parallèle — tu gaspilles 2 minutes de compute pour rien. Avec needs:, dès que le lint échoue, tout s'arrête. Tu as ton erreur en 30 secondes.

jobs:
  lint-and-typecheck:        # Job 1 — lance en premier
    ...

  test:
    needs: lint-and-typecheck  # Job 2 — attend le Job 1
    ...

  build:
    needs: [lint-and-typecheck, test]  # Job 3 — attend les deux
    ...

Pourquoi npm install -g pnpm@9.15.0 et pas l'action pnpm ?

⚠️
Leçon apprise à la dure
On utilisait pnpm/action-setup@v4. Fin avril 2026, cette action a commencé à causer des startup_failure — le workflow mourait avant même de lancer un seul job. Cause : le tag @v4 sur le marketplace GitHub pointait vers une version cassée. En utilisant npm install -g pnpm@9.15.0, on pin exactement la version qu'on veut, sans dépendre d'une action tierce.

✅ npm install -g pnpm@9.15.0

  • Version exacte pinned
  • Pas de dépendance externe
  • Toujours reproductible
  • Fonctionne partout

❌ pnpm/action-setup@v4

  • Tag peut bouger
  • Dépendance tierce
  • startup_failure sans raison
  • Difficile à déboguer

🏷️ pr.yml — Automatisation des PRs

Ce workflow s'occupe uniquement des automatisations liées aux Pull Requests. Il se déclenche uniquement sur l'événement pull_request.

Pourquoi séparer ça du CI ?

🚨
L'erreur qu'on a faite au départ
On avait mis le job label (avec actions/labeler@v5) directement dans ci.yml. Résultat : quand le labeler avait un problème, tout le CI échouait — même sur des pushs simples qui n'ont rien à voir avec les labels. Le CI devenait fragile à cause d'une fonctionnalité accessoire.

Ce que fait pr.yml

🏷️ Job : Auto Label

Lit le fichier labeler.yml et applique automatiquement les labels selon les fichiers modifiés :

  • Fichiers dans apps/frontend/ → domain: frontend
  • Fichiers dans apps/backend/ → domain: backend
  • Fichiers dans .github/ → domain: devops
🎯 Job : Assign Milestone

Lit le nom de la branche et assigne le milestone correspondant :

  • feature/phase-1a-* → Phase 1A
  • feature/phase-1b-* → Phase 1B
  • feature/phase-2-* → Phase 2 MVP

labeler.yml — La configuration du labeler

⚠️
Syntaxe v4 vs v5 — l'erreur classique
La version 5 de actions/labeler a changé sa syntaxe. L'ancienne syntaxe (any:) ne fonctionne plus. Il faut utiliser changed-files avec any-glob-to-any-file.

❌ Ancienne syntaxe (v4)

frontend:
  - any: ['apps/frontend/**']

backend:
  - any: ['apps/backend/**']

✅ Nouvelle syntaxe (v5)

"domain: frontend":
  - changed-files:
      - any-glob-to-any-file:
          - "apps/frontend/**"

"domain: backend":
  - changed-files:
      - any-glob-to-any-file:
          - "apps/backend/**"

Problèmes qu'on a rencontrés et solutions

1

startup_failure — softprops/action-gh-release@v2

L'action tierce causait un startup_failure à 0s. Solution : on l'a remplacée par gh release create — la CLI GitHub native, sans dépendance externe.

2

Heredoc avec ${{ }}

On écrivait le body de la PR avec un heredoc <<'EOF' contenant des expressions ${{ steps.version.outputs.version }}. GitHub tente de parser ces expressions au chargement du workflow, pas à l'exécution. Solution : on passe les valeurs via env: et on utilise des variables bash ${VERSION}.

3

Label release introuvable

On passait --label "release" mais le label s'appelle "type: release" avec le préfixe catégorie. La PR était créée mais sans label et l'erreur était silencieuse à cause du || echo "PR already exists". Solution : utiliser le bon nom de label.

🔒 codeql.yml — Sécurité

CodeQL est l'outil d'analyse statique de sécurité de GitHub. Il analyse le code pour trouver des vulnérabilités potentielles.

Ce qu'il détecte
  • Injection SQL
  • XSS (Cross-Site Scripting)
  • Exposition de données sensibles
  • Mauvaise gestion des erreurs
  • Dépendances avec CVE
Quand il tourne
  • Push sur main et develop
  • Pull requests
  • Tous les dimanches à minuit (cron)
schedule:
  - cron: "0 0 * * 0"
🧠
Pourquoi CodeQL n'a jamais eu de startup_failure ?
CodeQL n'utilise que des actions github/codeql-action — des actions officielles GitHub, jamais de tiers comme pnpm ou labeler. C'est pour ça qu'il était le seul workflow qui passait toujours. La leçon : moins on dépend d'actions tierces dans le CI critique, plus c'est stable.

🏗️ Séparation des responsabilités

C'est la règle d'or qu'on a appliquée. En entreprise, chaque workflow a une et une seule responsabilité.

Workflow Responsabilité Trigger Si ça casse
ci.yml Qualité du code push + pull_request 🚫 PR bloquée
pr.yml Automatisation PR pull_request uniquement ⚠️ Pas de labels (non bloquant)
release.yml Publication push release/* ⚠️ Release manuelle
codeql.yml Sécurité push + schedule ⚠️ Alerte sécurité
💡
Pourquoi c'est important
Si pr.yml a un bug (ex: labeler qui plante), le CI continue de fonctionner — les PRs sont toujours validées. L'automatisation des labels est une feature pratique mais jamais critique. En séparant, on protège ce qui est critique (CI) de ce qui est accessoire (labels).

🌿 GitFlow & Branches

Notre stratégie de branches définit qui peut faire quoi et où.

main — Production
↕ merge via release PR ou hotfix
develop — Intégration
feature/42-auth-page
↑ PR → develop
fix/43-login-crash
↑ PR → develop
release/0.2.0
↑ PR → main
hotfix/44-prod-bug
↑ PR → main + /sync

Convention de nommage des branches

PatternExempleDepuisMerge vers
feature/<numéro>-<desc> feature/29-landing-page develop develop
fix/<numéro>-<desc> fix/31-login-crash develop develop
hotfix/<numéro>-<desc> hotfix/32-auth-token main main + develop
release/<semver> release/0.1.0 develop main
🔢
Pourquoi le numéro d'issue dans le nom de branche ?
Le numéro d'issue dans le nom de branche sert deux choses : (1) traçabilité — on sait immédiatement à quelle issue cette branche correspond, (2) fermeture automatique — Closes #29 dans la PR body ferme l'issue quand la PR est mergée dans develop (la default branch).

Pourquoi develop est la default branch ?

⚠️
GitHub ferme les issues uniquement sur la default branch
Quand une PR avec Closes #42 est mergée, GitHub ferme l'issue seulement si la branche cible est la default branch. Si main était la default branch, les issues ne se fermeraient jamais automatiquement (les features mergent dans develop, pas main). On a donc changé develop comme default branch.

🛡️ Branch Protection Rules

Les branch protection rules sont des règles qui protègent tes branches importantes contre les pushs accidentels ou non vérifiés.

Configuration actuelle

main — Production
  • ✅ PR obligatoire
  • ✅ 1 reviewer requis
  • ✅ CI doit être vert
  • ✅ Code owner review
  • ❌ Pas de push direct
develop — Intégration
  • ✅ PR obligatoire
  • ✅ 1 reviewer requis
  • ✅ CI doit être vert
  • ⭐ enforce_admins: false
  • → Tu peux bypasser comme admin

Le bouton "Merge without waiting for requirements"

Puisque enforce_admins: false, en tant qu'admin du repo, tu vois ce bouton sur tes propres PRs. Il te permet de merger même sans reviewer — c'est ton bypass personnel. Les autres contributors ne le voient pas.

🏢
En entreprise avec plusieurs devs
Quand d'autres personnes rejoignent le projet, CODEOWNERS te désigne automatiquement comme reviewer sur toutes leurs PRs. Tu reçois une notification GitHub et tu peux approuver ou demander des modifications. Eux ne peuvent pas bypasser — ils doivent attendre ta review.

CODEOWNERS

# .github/CODEOWNERS
# * signifie "tous les fichiers"
# @AdamDjo sera automatiquement ajouté comme reviewer
# sur toute PR qui touche n'importe quel fichier

* @AdamDjo

🐛 Erreurs rencontrées et solutions

Voici un catalogue des vraies erreurs qu'on a eu et comment les identifier / résoudre.

startup_failure à 0s

startup_failure 0 jobs lancés, 0 logs disponibles

Cause : GitHub ne peut pas parser ou résoudre le workflow avant de le lancer. Cela peut venir de :

  • Une action tierce avec un tag cassé (pnpm/action-setup@v4)
  • Une action tierce qui n'est pas dans la whitelist (softprops/action-gh-release@v2)
  • Une erreur de syntaxe YAML invisible

Comment diagnostiquer : compare avec le dernier commit qui marchait — git diff <bon-commit>..HEAD -- .github/workflows/

ERR_PNPM_OUTDATED_LOCKFILE

failure Install dependencies — exit code 1

Cause : Le pnpm-lock.yaml n'est pas synchronisé avec package.json. Ça arrive quand on résout un merge conflict en prenant le mauvais côté du lockfile.

Solution : pnpm install en local, puis commit le pnpm-lock.yaml mis à jour.

Release.tag_name already exists

failure Create GitHub Release — HTTP 422

Cause : Le workflow release a déjà créé le tag lors d'un run précédent. Les pushs suivants sur la même branche retentent de créer le même tag.

Ce que ça veut dire concrètement : ce n'est pas un vrai problème — la release est déjà publiée. C'est juste le workflow qui ne sait pas qu'il a déjà fait son travail. Sûr de merger.

✨ Bonnes pratiques enterprise

🎯 Un workflow = une responsabilité

Ne jamais mélanger CI (qualité) avec automation PR (labels). Si l'automation plante, le CI reste vert.

📌 Pin les versions exactes

pnpm@9.15.0 et non pas pnpm@latest. Un tag @v4 peut pointer vers n'importe quoi demain.

🔧 Préférer les outils natifs

gh release create plutôt que softprops/action-gh-release. La CLI GitHub est maintenue par GitHub, jamais cassée.

⚡ Fail fast avec needs:

Toujours chaîner les jobs critiques. Si le lint échoue, ne pas lancer les tests — économise du temps et de l'argent.

🔕 Masquer les erreurs avec soin

|| echo "PR already exists" masque les vraies erreurs. Utiliser seulement quand l'erreur est vraiment attendue et non-critique.

🌿 Default branch = develop

Dans un GitFlow, develop est la branche d'intégration. GitHub ferme les issues automatiquement uniquement sur la default branch.
